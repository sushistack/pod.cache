<summary>🚀 왜 멀티스레딩인가 반응성과 성능의 비밀</summary>


### 멀티쓰레딩이 필요한 이유?
1. 반응성
2. 성능 향상

단일 스레드 vs 멀티 스레딩
- 단일: 한명이 사용하면 다른 사람 기다림 (화장실) => 멈춤이 발생
- 멀티: 사용자 별로 처리하여 => 동시 응답

### 동시성 vs 병렬성
- 동시성: 단일 코어 환경에서도 여러 스레드를 스위칭하며 실행 (착시에 가까움)
- 병렬성: 다중 코어에서 실제로 병렬성 가능 (진짜 동시에 실행)


### 성능 향상 이점
- 복잡한 작업의 빠른 완료
- 처리량 증가
- 인프라 비용 절감


### 프로세스 vs 스레드
- OS: 하드웨어와 CPU에 대한 추상화 계층 제공
- 프로세스: OS 가 프로그램 파일을 메모리에 올려 로드하고 실행 인스턴스를 생성 (=> 이것이 프로세스 혹은 애플리케이션의 "컨텍스트")
- 또한 프로세스는 각각 격리되어 있어 다른 프로세스에 대한 영향이 없음.


### 프로세스 구성 요소
- 메타데이터 (Metadata): 프로세스 ID(PID), 열린 파일 목록 등
- 코드 (Code): 실행될 프로그램 명령어
- 힙 (Heap): 동적으로 할당되는 데이터 영역
- 스택 (Stack): 지역 변수 및 함수 호출 정보 (최소 하나의 스레드를 통해 존재)
    ㄴ 스택은 엄밀히 말하면 스레드의 고유 영역에 속하지만, 프로세스 전체의 메모리 공간 내에 존재
- 최소 하나의 스레드 (Main Thread): 모든 프로세스는 하나 이상의 스레드를 가짐

### 스레드
- 스레드(Thread)는 프로세스 내에서 실제 코드를 실행하는 흐름의 단위
- 멀티스레드 애플리케이션에서는 각 스레드가 고유한 요소를 가지면서도, 프로세스의 일부 리소스를 공유



### 스레드 고유 영역
- 스택
- 명령어 포인터


### 스레드가 공유하는 영역
- 코드
- 힙
- 파일


| 구성 요소        | 프로세스 (공유)    | 스레드 (고유)      | 설명   |
| :--------------- | :----------------- | :----------------- | :------- |
| **메타데이터**   | ✅                 | ❌                 | PID, 열린 파일 등    |
| **코드**         | ✅                 | ❌                 | 프로그램 명령어 |
| **힙 (Heap)**    | ✅                 | ❌                 | 동적으로 할당되는 데이터   |
| **스택 (Stack)** | ❌                 | ✅                 | 지역 변수 및 함수 호출 정보 (각 스레드마다 독립)        |
| **명령어 포인터** | ❌                 | ✅                 | 다음 실행할 명령어의 주소 (각 스레드마다 독립)          |
