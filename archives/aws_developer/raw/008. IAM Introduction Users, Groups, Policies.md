# 008. 🚀 001. AWS IAM 딥다이브: 클라우드 접근 제어의 핵심

AWS IAM(Identity and Access Management)은 클라우드 자원에 대한 접근 권한을 안전하게 관리하는 글로벌 서비스입니다. 사용자, 그룹, 정책을 통해 개별 및 조직 단위의 권한을 체계적으로 부여할 수 있으며, '최소 권한 원칙'을 적용하여 보안을 강화하고 비용 효율적인 운영을 달성합니다.

## 1. 서론: AWS IAM의 중요성

클라우드 환경에서는 수많은 사용자와 애플리케이션이 다양한 리소스에 접근합니다. 이러한 접근을 안전하고 효율적으로 제어하는 것은 클라우드 보안의 가장 기본적인 토대입니다. AWS IAM(Identity and Access Management)은 바로 이러한 요구사항을 충족시키기 위해 설계된 AWS의 핵심 서비스입니다.

AWS 계정을 처음 생성할 때 우리는 '루트(Root) 계정'을 생성하게 됩니다. 이 루트 계정은 AWS 계정의 모든 서비스와 리소스에 대한 무제한적인 접근 권한을 가지며, 매우 강력한 권한을 보유하고 있습니다. 하지만 이러한 강력한 권한은 그만큼 위험성도 높습니다. 따라서 루트 계정은 초기 설정 외에는 사용을 지양하고, 대신 IAM을 통해 개별 사용자 계정을 생성하여 사용하는 것이 AWS의 모범 사례입니다. IAM은 전역 서비스(Global Service)로, 특정 리전에 종속되지 않고 AWS 전체에서 사용자 및 권한을 관리합니다.

## 2. 본론: IAM의 핵심 구성 요소와 원칙

IAM은 크게 사용자(Users), 그룹(Groups), 그리고 정책(Policies)의 세 가지 핵심 구성 요소로 이루어져 있습니다. 이들을 조합하여 AWS 리소스에 대한 접근 권한을 세밀하게 제어할 수 있습니다.

### 2.1. 사용자 (Users)

IAM 사용자는 AWS 계정 내에서 특정 개인이나 애플리케이션을 나타냅니다. 각 사용자는 고유한 자격 증명(ID 및 비밀번호, 또는 Access Key/Secret Access Key)을 가지며, AWS 콘솔 또는 프로그래밍 방식으로 AWS 서비스에 접근할 수 있습니다.

### 2.2. 그룹 (Groups)

그룹은 IAM 사용자들을 효율적으로 관리하기 위한 논리적인 집합입니다. 여러 사용자에게 동일한 권한을 부여해야 할 때, 각 사용자에게 개별적으로 권한을 부여하는 대신, 그룹에 권한을 부여하고 해당 사용자들을 그룹에 추가함으로써 관리의 복잡성을 줄일 수 있습니다.

**그룹의 주요 특징:**

*   **사용자만 포함**: 그룹은 다른 그룹을 포함할 수 없으며, 오직 IAM 사용자만 포함할 수 있습니다.
*   **다중 그룹 소속**: 한 사용자는 여러 그룹에 소속될 수 있습니다.
*   **유연한 관리**: 조직의 역할에 따라 '개발자 그룹', '운영자 그룹', '감사자 그룹' 등을 생성하여 관리 효율성을 높일 수 있습니다.

**예시: 조직 내 사용자 및 그룹 구성**

| 사용자  | 개발자 그룹 | 운영 그룹 | 감사 그룹 |
| :------ | :---------- | :-------- | :-------- |
| Alice   | O           | X         | X         |
| Bob     | O           | X         | X         |
| Charles | O           | X         | O         |
| David   | X           | O         | O         |
| Edward  | X           | O         | X         |
| Fred    | X           | X         | X         |

위 예시에서 Charles와 David는 각각 '개발자 그룹'과 '운영 그룹'에 속하면서도, 동시에 '감사 그룹'에도 속해 여러 그룹의 권한을 가질 수 있습니다. Fred와 같이 어느 그룹에도 속하지 않는 사용자도 존재할 수 있습니다.

### 2.3. 정책 (Policies)

정책은 AWS 리소스에 대한 접근 권한을 정의하는 JSON(JavaScript Object Notation) 문서입니다. 이 정책은 IAM 사용자, 그룹, 또는 역할(Role)에 연결되어 해당 엔티티가 수행할 수 있는 작업을 명시합니다.

**정책의 구조:**

정책은 `Statement` 배열을 포함하며, 각 `Statement`는 `Effect`, `Action`, `Resource` 등의 요소를 가집니다.

*   `Effect`: `Allow` (허용) 또는 `Deny` (거부)
*   `Action`: 허용 또는 거부할 AWS 서비스의 특정 작업 (예: `ec2:RunInstances`, `s3:GetObject`)
*   `Resource`: 작업이 적용될 AWS 리소스 (예: `arn:aws:s3:::my-bucket/*`)

**정책 예시:**

다음은 EC2, Elastic Load Balancing, CloudWatch 서비스에 대한 `Describe` 작업(정보 조회)을 허용하는 IAM 정책의 예시입니다.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "ec2:Describe*",
        "elasticloadbalancing:Describe*",
        "cloudwatch:*"
      ],
      "Resource": "*"
    }
  ]
}
```

이 정책은 개발자 그룹에 연결될 경우, 해당 그룹에 속한 모든 개발자가 EC2 인스턴스 정보, 로드 밸런서 정보, CloudWatch 지표 등을 조회할 수 있도록 허용합니다.

### 2.4. 최소 권한 원칙 (Principle of Least Privilege)

IAM을 효과적으로 사용하는 가장 중요한 원칙은 '최소 권한 원칙'입니다. 이 원칙은 사용자나 애플리케이션에 필요한 최소한의 권한만 부여해야 한다는 것을 의미합니다.

*   **보안 강화**: 불필요한 권한을 제한하여 잠재적인 보안 위험(예: 권한 남용, 계정 탈취 시 피해 최소화)을 줄일 수 있습니다.
*   **비용 절감**: 의도치 않은 서비스 생성이나 리소스 수정으로 인한 불필요한 비용 발생을 방지합니다.

예를 들어, 특정 사용자가 단 세 가지 서비스에만 접근해야 한다면, 해당 세 서비스에 대한 권한만 명시적으로 부여해야 합니다. 모든 서비스에 대한 접근 권한을 부여하는 것은 매우 위험하며 권장되지 않습니다.

## 3. 결론: 안전한 클라우드 운영을 위한 IAM의 역할

AWS IAM은 클라우드 환경에서 사용자와 리소스 간의 접근 권한을 관리하는 데 필수적인 서비스입니다. 루트 계정의 사용을 최소화하고, IAM 사용자 및 그룹을 통해 역할을 분리하며, '최소 권한 원칙'에 따라 세분화된 정책을 적용함으로써 우리는 보다 안전하고 효율적인 클라우드 운영 환경을 구축할 수 있습니다.

이처럼 IAM의 기본적인 개념과 구성 요소를 이해하는 것은 AWS 클라우드 여정의 첫걸음이자 가장 중요한 단계입니다. 다음 강의에서는 IAM 사용자 및 그룹을 직접 생성하고 권한을 부여하는 실습을 통해 오늘 배운 개념들을 직접 경험해보도록 하겠습니다.
